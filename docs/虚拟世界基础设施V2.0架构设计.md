# 虚拟世界基础设施 V2.0 架构设计

**数学模型与LLM深度融合方案**

---

## 1. 核心洞察

### 1.1 为什么单靠数学模型不够？

| 数学模型的优势 | 数学模型的局限 |
|---------------|---------------|
| 确定性计算 | 无法处理语义模糊 |
| 可复现性 | 缺乏创造力 |
| 高效执行 | 无法理解上下文 |
| 参数可控 | 输出冰冷、机械 |

### 1.2 为什么单靠LLM不够？

| LLM的优势 | LLM的局限 |
|----------|----------|
| 语义理解 | 结果不可控 |
| 创造力 | 计算成本高 |
| 上下文理解 | 缺乏一致性 |
| 自然输出 | 无法保证逻辑严密 |

### 1.3 结论：必须深度融合

**数学模型 = 骨架（结构、逻辑、约束）**
**LLM = 血肉（语义、创造、表达）**

## 3. 核心设计原则

### 3.1 双向绑定原则

**数学模型 ⇄ LLM 不是单向的**

```
数学模型 ──提供约束──► LLM
    ▲                    │
    │                    │
    └────返回影响────────┘
```

**例子**：
```
数学模型：饥荒强度0.8，角色精神失真度0.3
    ↓ 约束
LLM：生成"角色决定偷偷储备粮食"的行动描述
    ↓ 影响
数学模型：资源消耗-5%/tick，被发现概率+10%，精神失真度+0.05
```

### 3.2 分层职责原则

| 层级 | 数学模型职责 | LLM职责 |
|------|-------------|---------|
| 世界层 | 四大维度存量计算、物理环境演化 | 作为“世界翻译官”，将自然语言解析为对四大维度的张量冲击（OpenEvent） |
| 角色层 | 材质力学参数计算、多维主观应力计算、相变判定 | 决定是否进行合金化注入、生成行动描述与内心独白 |
| 交互层 | 社会流体力学演算、雷诺数计算、宏观相变 | 演绎群体情绪极化、生成多方冲突的叙事细节 |

### 3.3 语义锚点原则

**每个数学参数都有对应的语义锚点**

| 数学参数 | 语义锚点 |
|---------|---------|
| 精神失真度 0.3 | "偶尔出现不切实际的想法" |
| 精神失真度 0.7 | "分不清现实与幻想" |
| 创伤累积度 0.5 | "过去的阴影挥之不去" |
| 决策"隐忍" | "咬紧牙关，强忍不发" |

### 4.2 角色层：有情世间引擎 + LLM

#### 4.2.1 七步演算的LLM增强

| 步骤 | 数学模型 | LLM增强 |
|------|---------|--------|
| **触 (Contact)** | 感知与多维主观应力计算（结合执念矩阵） | （无，纯数学） |
| **缘 (Conditioning)** | 情境缓冲与历史应力集中点 ($K_t$) 放大 | （无，纯数学） |
| **受 (Sensation)** | 弹性应变计算、塑性形变累积 | 接收数学状态，准备后续生成 |
| **想 (Cognition)** | 动机生成、相变触发判定 | **合金化判定**：生成注入元素 (C/S/Cr/Ni/P/Mn等) 及原因 |
| **行 (Conflict)** | 驱动力与阻力的内耗计算 ($F_{int}$) | 心理挣扎描述 |
| **识 (Decision)** | 基于净动力的五态决策生成 (EXPLODE等) | 具体行动方案生成、身体语言、情绪 |
| **业 (Karma)** | 多维业力写回 (对四大维度损耗)、状态持久化 | 生成决策产生的完整叙事日志 |

#### 4.2.2 核心数据结构

```python
@dataclass
class CharacterState:

    toughness: float  # 韧性

    expected_resources: Dict[ResourceDim, float]

@dataclass
class Decision:

    intensity: float               # 强度
    confidence: float              # 置信度
    
    # LLM输出
    action_description: str        # 具体行动描述
    inner_monologue: str           # 内心独白
    body_language: str             # 身体语言

    # 反馈到数学模型
```

#### 4.2.3 演算流程

```python
class CharacterLayer:
    def process_event(self, character: Character, event: Event):
        # Step 1: 触（感知与应力）
        contact = self.math_contact(character, event)
        
        # Step 2: 缘（情境与应力集中）
        conditioning = self.math_conditioning(character, contact)
        
        # Step 3: 受（应变计算）
        sensation = self.math_sensation(character, conditioning)
        # LLM生成情感描述
        sensation.emotion_description = self.llm_engine.describe_emotion(
            character, sensation
        )
        
        # Step 4: 想（认知与合金化）
        cognition = self.math_cognition(character, sensation)
        # LLM判定合金化并生成内心独白
        cognition.alloy_injection, cognition.inner_monologue = self.llm_engine.judge_alloy_and_voice(
            character, cognition
        )
        
        # Step 5: 行（内耗）
        conflict = self.math_conflict(character, cognition)
        # LLM描述心理挣扎
        conflict.struggle_description = self.llm_engine.describe_struggle(
            character, conflict
        )
        
        # Step 6: 识（决策）
        decision = self.math_decision(character, conflict)
        # LLM根据五态生成具体行动
        decision.action_description = self.llm_engine.generate_action(
            character, decision
        )
        
        # Step 7: 业（状态演化与写回）
        karma = self.math_karma(character, decision)
        # LLM生成后果叙事日志
        karma.consequence_description = self.llm_engine.describe_consequence(
            character, karma
        )
        
        # 反馈：LLM输出影响数学参数
        self.apply_feedback(character, decision, karma)
        
        return decision
```

## 5. 记忆系统：基于显著度衰减的情景缓存 (V0 原型)

*注意：当前工程版本实现的是一个**高权重日志片段池 (Salience-Decay Episodic Cache)**，主要用于在有限的 Token 上下文中保留最深刻的记忆。尚未引入重量级的 Vector DB 或 Top-K 语义检索。*

### 5.1 记忆结构

```python
@dataclass
class EpisodicMemory:
    tick: int
    description: str            # 事件描述与自身反应
    salience: float             # 记忆重要度(基于物理应力和塑性形变)
```

### 5.2 记忆缓存淘汰机制

当前系统不使用复杂的向量相似度检索，而是采用优先队列机制：
- 每个角色最多保留 N 条（如 10 条）近期记忆。
- 记忆的权重随时间衰减：`current_weight = salience * exp(-k * time_passed)`。
- 当容量超限时，淘汰当前权重最低的一条。
- 在构建 LLM Prompt 时，直接送入最近/最深刻的 M 条记忆（如 3 条），以及针对特定实体的专属图谱记忆。

### 6.2 调用优化

```python
class LLMCallStrategy:
    def should_call_llm(self, event: Event, character: Character) -> bool:
        """判断是否需要调用LLM"""
        
        # 重大事件必须调用
        if event.intensity > 0.7:
            return True
        
        # 用户正在查看
        if character.is_being_watched:
            return True
        
        # 关键决策点
        if character.state_change_threshold():
            return True
        
        # 随机采样（保持一定比例的LLM调用）
        if random.random() < 0.1:
            return True
        
        return False
```

### 6.3 批量处理

```python
class BatchLLMProcessor:
    def process_batch(self, characters: List[Character], 
                     events: List[Event]) -> List[Decision]:
        """批量处理，减少LLM调用次数"""
        
        # 1. 按事件类型分组
        grouped = self.group_by_event_type(events)
        
        # 2. 同类事件共享LLM调用
        results = []
        for event_type, chars_events in grouped.items():
            # 一次LLM调用，生成模板
            template = self.llm_engine.generate_event_template(event_type)
            
            # 数学模型个性化
            for char, event in chars_events:
                decision = self.personalize(template, char, event)
                results.append(decision)
        
        return results
```

## 8. 执行语义、并发与持久化 (CQRS & Tick隔离)

为了解决多角色并发演算时的读写竞争问题，并提供工业级的持久化方案，系统采用**Tick隔离（双缓存机制）**与**CQRS（命令查询职责分离）**架构。

### 8.1 Tick 隔离与双缓存机制 (Double Buffering)

*注意：当前工程版本（V10.0）主要在 `engine.py` 的主循环层面实现了宏观的 `pending_writebacks` 回写统一，但 `SentientEngine` 内部在单线程演算时仍会直接修改角色状态（如 `plastic_strain`）。严格的“读只读快照，绝对禁止原地修改”的工业级隔离将在未来的并发版本中完整实现。*

在任意 Tick $N$ 中，角色的感知与演算必须是确定性的，不受其他角色正在执行的动作影响。

**执行语义规则：**
1. **读状态锁定**：在 Tick $N$ 的所有并发演算（感知、决策）中，角色只能读取只读的“状态快照 $State_N$”。
2. **写回隔离**：角色在“业 (Karma)”阶段产生的影响（Impacts）不能直接修改 $State_N$，而是作为增量命令（Command）被放入一个**全局事务池 (Transaction Pool)**。
3. **合并结算**：当所有角色的并发演算结束后，**事务解析器 (Karma Resolver)** 介入，将所有增量命令合并，并统一施加于 $State_N$，生成新的只读快照 $State_{N+1}$。

```python
class WorldEngine:
    def tick(self):
        # 1. 冻结当前状态快照 (Read-only)
        state_N = self.get_frozen_snapshot()
        pending_impacts = []
        
        # 2. 并发执行所有角色的七步演算
        for char in self.characters: # 可放入线程池/协程池
            decision = char.process_pipeline(state_N)
            pending_impacts.extend(decision.karma_impacts)
            
        # 3. 冲突合并与规则结算
        resolved_impacts = self.karma_resolver.merge(pending_impacts)
        
        # 4. 生成新状态
        self.state = self.apply_impacts(state_N, resolved_impacts)
```

### 8.2 冲突合并规则 (Karma Resolution)

当多个角色对同一资源（如同一个村庄的粮食）产生冲突影响时：
- **资源透支**：如果总掠夺量超过存量，按角色权力/武力权重分配。
- **状态抵消**：正向情绪广播与负向情绪广播在同一区域相遇，按强度衰减后取净值。

### 8.3 持久化工程：CQRS 与事件溯源 (Event Sourcing)

*注意：当前工程版本（V10.0）通过 `_dump_log` 将每个 Tick 的 `OpenEvent` 和 `KarmaWriteback` 写入 JSONL 文件，已具备 Event Sourcing 原型的能力，但并未接入实际的数据库及真正的 Snapshot 恢复机制。以下为标准工程规范：*

**痛点**：如果只保存最终状态，无法追溯角色心理变化的完整历史，且容易在意外宕机时损坏数据。

**方案**：
1. **命令端 (Command/Write)**：引擎运行时，不频繁执行数据库的 UPDATE。每一个产生的 `OpenEvent` 和 `KarmaImpact` 都被追加写入（Append-only）到**事件日志 (Event Log)** 中（如 Kafka / SQLite Append）。这是绝对的 Single Source of Truth。
2. **查询端 (Query/Read)**：内存中维护“热状态 (Hot State)”用于快速读取和演算。
3. **快照机制**：每隔 $M$ 个 Tick，将内存热状态做一次全量 Snapshot 存入数据库。恢复时只需加载最新 Snapshot，然后重放其后的 Event Log。

```python
# Event Sourcing 写入示例
def persist_tick(tick_id, impacts):
    db.execute("INSERT INTO event_logs (tick, data) VALUES (?, ?)", 
               (tick_id, json.dumps(impacts)))

# 系统崩溃后的恢复
def recover_world():
    latest_snapshot = db.get_latest_snapshot()
    logs = db.get_event_logs_after(latest_snapshot.tick)
    
    world = World(latest_snapshot)
    for log in logs:
        world.apply_impacts(log.impacts) # 确定性重放
    return world
```

## 10. 工程关键点：LLM输出的结构化

### 10.1 全局枚举真相表 (Single Source of Truth)

为彻底封死文档和代码中的“接口级不一致”，现确立以下**唯一枚举真相表**。所有代码实现、提示词和持久化数据必须且只能使用以下枚举值：

#### 决策类型 (DecisionType)
这是七步管线中“识”步骤的唯一输出接口。
| 枚举值 | 含义 | 典型映射场景 |
|------|------|---------|
| `EXPLODE` | 爆发 | 极端对抗、马氏体屠杀、奥氏体牺牲、渗碳体傲慢掠夺 |
| `ACT` | 行动 | 正常交涉、劳动、常规社交行为 |
| `HESITATE` | 犹豫 | 内耗严重时的僵局、退缩 |
| `SUPPRESS` | 隐忍 | 压抑怒火、为了保全其他资源而妥协 |
| `BREAKDOWN` | 崩溃 | FRACTURE态直接输出，丧失理智的混乱 |

#### 情感标签 (EmotionType)
| 标签 | 含义 | 标签 | 含义 |
|------|------|------|------|
| `NEUTRAL` | 平静 | `GRATEFUL` | 感激 |
| `FEAR` | 恐惧 | `DESPAIR` | 绝望 |
| `ANGER` | 愤怒 | `COLD` | 冷漠 (高刚度专属) |
| `SADNESS` | 悲伤 | `MANIC` | 狂躁 (高毒性专属) |
| `JOY` | 喜悦 | `HOLLOW` | 空洞 (断裂前夕) |
| `HOPEFUL` | 希望 | | |

#### 合金元素 (AlloyElement)
| 元素 | 含义 | 影响 |
|------|------|---------|
| `C` | 碳 | 变硬、变脆、变傲慢 |
| `S` | 硫 | 增加毒性、大幅降低崩溃阈值 |
| `Cr` | 铬 | 意志坚定、提高屈服点 |
| `Ni` | 镍 | 变得心软、极大提升韧性 |
| `P` | 磷 | 思维混乱、双重下降防线 |
| `Mn` | 锰 | 变得坚忍耐磨 |

#### 物理应力类型 (StressType)
| 类型 | 含义 | 典型事件 |
|------|------|---------|
| `COMPRESSIVE` | 压缩 | 饥荒、贫穷、持续压迫 |
| `TENSILE` | 拉伸 | 意外横财、灾后重建、爱意 |
| `SHEAR` | 剪切 | 战乱、背叛、突发暴力 |
| `CORROSION` | 腐蚀 | 瘟疫、谣言、慢性精神折磨 |

### 10.2 核心痛点

**LLM文本到数学参数的"反向解析"极其脆弱**

| 问题 | 表现 |
|------|------|
| 格式错误 | JSON解析失败、字段缺失 |
| 数值发散 | 有时扣0.1，有时扣0.9 |
| 语义漂移 | 同样的输入产生不一致的输出 |

### 10.3 解决方案：强制结构化输出

**原则：不让LLM直接输出产生逻辑分支和冲击的浮点数，而是输出标签，由代码映射**

*注：合金浓度 (`concentration`) 由于只涉及角色微观物理面板的缓慢漂移，不直接决定系统主循环分支，因此作为唯一的连续值 (0.01~0.05) 例外保留。其余所有主干决策与情感必须是严格标签。*

```python
# ❌ 错误做法：让LLM直接输出数值
LLM输出: {"B_intm": -0.1, "trust": -0.3}  # 数值发散、不可控

# ✅ 正确做法：让LLM输出标签，代码映射
LLM输出: {"conflict_level": "SEVERE", "emotion": "ANGER"}

# 代码映射
MAPPING = {
    "SEVERE": -0.3,
    "MODERATE": -0.15,
    "MILD": -0.05,
    "NONE": 0.0
}

impact = MAPPING[llm_output["conflict_level"]]
```
